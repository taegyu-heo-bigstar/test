📝 백엔드 기초 개념 및 프로젝트 아키텍처 제안

1. API와 REST API의 이해

백엔드 서버의 역할을 이해하기 쉽게 '식당'에 비유해 보겠습니다.

API 서버란?
API 서버는 클라이언트의 요청을 정해진 인터페이스 규칙에 따라 받아 처리하고, 그 결과를 반환하는 서버입니다. 식당으로 치면 손님의 주문을 받고, 주방과 연결해 결과를 내어주는 **'점원'**에 가깝습니다.

REST API란?
REST는 웹을 위한 아키텍처 스타일입니다. 실무에서는 보통 HTTP 메서드와 URI를 이용해 자원(Resource)을 조회·생성·수정·삭제하는 형태로 구현합니다. 즉, REST API는 클라이언트와 서버가 자원을 중심으로 통신하도록 정리한 방식이라고 볼 수 있습니다. 

예시
예를 들어 특정 사용자의 정보를 조회할 때는 다음과 같이 표현할 수 있습니다.

GET /users/{user_id}

이처럼 클라이언트는 무엇을 원하는지를 URI와 HTTP 메서드로 표현하고, 서버는 그 요청을 처리해 JSON 등의 형태로 응답합니다.

2. 개발 프레임워크 비교와 FastAPI 추천

이번 프로젝트는 단순 웹페이지가 아니라, 모바일/태블릿 클라이언트가 필기 이미지를 업로드하고, 서버가 이를 받아 AI 분석을 수행한 뒤 결과 리포트를 반환하는 구조를 가정하고 있습니다. 따라서 프레임워크 비교도 단순 인기보다 다음 기준으로 보는 것이 더 적절합니다.

Python 기반 AI 처리와의 연계성

API 문서화와 요청/응답 검증 편의성

팀 학습 난이도와 개발 생산성

구조적 확장성

성능과 유지보수성


📊 주요 프레임워크 비교

프레임워크	주 언어	공식적으로 드러나는 성격	프로젝트 관점 평가

FastAPI	Python	Python 타입 힌트 기반 API 프레임워크이며, OpenAPI 기반 자동 문서화와 async/await 기반 비동기 처리를 지원함. 	가장 추천. AI 코드와 같은 Python 생태계로 묶기 좋고, 문서화·검증·API 개발 속도의 균형이 좋음.
Flask	Python	경량 WSGI 웹 프레임워크로, 빠르게 시작하기 쉽고 유연성이 큼. 	가볍고 배우기 쉽지만, 인증·검증·문서화 등을 직접 더 많이 구성해야 해서 팀 프로젝트에서는 설정 부담이 커질 수 있음.
Django	Python	“batteries-included” 철학의 프레임워크이며, ORM·인증 시스템·관리 기능 등 기본 제공 요소가 많음. 	기능이 강력하지만, 이번처럼 API 중심 + AI 연동 구조에는 다소 무겁게 느껴질 수 있음.
Express	JavaScript / Node.js	빠르고 최소주의적인 Node.js 웹 프레임워크로, 미들웨어 기반 유연성이 큼. 	자유도는 높지만, AI 분석 서버를 Python으로 둘 경우 언어가 분리되어 전체 구조가 이원화될 가능성이 큼.
NestJS	TypeScript / Node.js	효율적이고 확장 가능한 Node.js 서버 애플리케이션을 위한 구조화된 프레임워크이며, TypeScript 중심 설계를 가짐. 	대규모 팀 개발에는 강하지만, Python AI 스택과 바로 맞물리는 점에서는 FastAPI보다 불리함.
Gin	Go	고성능 Go HTTP 웹 프레임워크로, 빠른 라우팅과 생산성을 강조함. 	성능은 우수하지만, 팀이 Go에 익숙하지 않다면 학습 비용이 생기고 AI 서버와의 언어 일관성도 떨어짐.


✅ 왜 FastAPI를 가장 추천하는가

1. AI/딥러닝 서버와 언어 생태계를 통일하기 쉽습니다.
필체 분석, 이미지 전처리, 추론 로직은 Python 생태계에서 구현하는 경우가 많습니다. FastAPI도 Python 기반이므로, 백엔드 API 계층과 AI 처리 계층을 같은 언어 계열로 맞추기 쉽습니다. 


2. 자동 문서화가 강력합니다.
FastAPI는 OpenAPI 기반 자동 문서화와 기본 내장 문서 UI를 제공합니다. 즉, 프론트엔드·백엔드 협업에서 API 명세 관리가 상대적으로 쉬워집니다. 


3. 요청/응답 검증이 편리합니다.
FastAPI는 타입 선언과 모델 기반으로 요청/응답 구조를 문서화하고 검증하는 흐름이 잘 갖춰져 있습니다. 입력 형식이 분명한 분석 API에 적합합니다. 


4. 비동기 처리에 자연스럽게 대응할 수 있습니다.
이미지 업로드, 결과 대기, 내부 분석 서버 호출처럼 I/O 대기가 있는 구조에서 async/await 지원은 실용적입니다. 


5. Flask보다 체계적이고, Django보다 가볍습니다.
Flask는 매우 가볍지만 많은 부분을 직접 설계해야 하고, Django는 다양한 기능을 제공하는 대신 API 서버만 두기에는 다소 무거울 수 있습니다. FastAPI는 그 중간에서 API 중심 프로젝트에 특화된 균형점에 가깝습니다. 



결론

여러 프레임워크를 비교하면, 이번 프로젝트에서는 FastAPI를 가장 추천할 수 있습니다.
핵심 이유는 다음 네 가지입니다.

Python 기반 AI 처리와의 자연스러운 연결

자동 API 문서화

강한 데이터 검증 구조

API 서버 중심 개발에 적합한 무게감


즉, 필체 교정 앱처럼 “이미지 업로드 → 서버 처리 → AI 분석 → 결과 반환”이 핵심인 구조에는 FastAPI가 가장 실용적입니다.

3. 필체 교정 앱을 위한 백엔드 구조 제안

단순한 클라이언트-서버 통신만 고려한다면 일반적인 REST API 구조만으로도 충분합니다.
그러나 이 프로젝트는 다음과 같은 흐름을 가집니다.

이미지 업로드

필체 분석

오차율 계산

결과 리포트 생성 및 반환


이 경우 일반 서비스 서버와 딥러닝 처리 서버를 분리하는 편이 더 적절합니다. FastAPI는 외부 API 계층을 구성하는 데 적합하고, 내부 AI 처리 계층 역시 Python 기반으로 자연스럽게 연결하기 좋습니다. 

🏗️ 제안하는 서버 아키텍처

graph LR
    Client[📱 클라이언트] -- REST API --> AppServer[💻 일반 서버 (FastAPI)]
    AppServer -- gRPC or Internal API --> DLServer[🧠 딥러닝 서버]

구조 설명

클라이언트

사용자가 모바일 또는 태블릿에서 필체를 입력하거나 이미지를 업로드합니다.

로그인, 분석 요청, 결과 리포트 조회 등을 수행합니다.


일반 서버 (FastAPI)

인증, 사용자 관리, 요청 수신, 파일 저장, 결과 반환 등 서비스 로직 전반을 담당합니다.

클라이언트와 직접 통신하는 주 백엔드 서버입니다.

필요 시 딥러닝 서버에 분석 요청을 보냅니다.

획 정보 추출

분석 결과 생성



이렇게 분리하면 일반 웹 기능과 AI 추론 기능을 나눌 수 있어 유지보수와 확장성이 좋아집니다. 추후 모델 교체나 성능 개선이 필요할 때도 딥러닝 서버를 별도로 조정하기 쉬워집니다.

4. REST API, gRPC, MQTT, WebSocket 비교

프로젝트의 통신 방식은 하나만 고르는 문제가 아니라, 어느 계층에서 어떤 문제를 해결할 것인가로 나누어 보는 것이 더 정확합니다.

REST는 웹 자원을 다루는 아키텍처 스타일입니다. 

gRPC는 .proto 기반 서비스 정의와 원격 메서드 호출을 중심으로 하는 RPC 프레임워크이며, Protocol Buffers와 HTTP/2를 활용할 수 있습니다. 

WebSocket은 브라우저와 서버 사이에 양방향 상호작용 세션을 여는 기술입니다. 

MQTT는 가볍고 단순한 publish/subscribe 메시징 프로토콜로, IoT와 제약 환경에 적합합니다. 


📊 통신 방식 비교

방식	통신 모델	공식적 특징	장점	단점	이 프로젝트에서의 적합성

REST API	요청-응답	자원 중심, 웹 아키텍처 스타일, 일반적으로 HTTP 기반 구현 	구조가 직관적이고 모바일/웹 클라이언트와 연동이 쉬움	실시간 양방향 통신에는 비효율적일 수 있음	매우 높음
gRPC	RPC, 요청-응답 + 스트리밍	.proto 서비스 정의, Protocol Buffers 사용 가능, HTTP/2 및 스트리밍 지원 	서버 간 통신에 효율적이고 계약이 명확함	브라우저 클라이언트에서 직접 사용하기는 상대적으로 불편함	서버 간 통신에 매우 높음
WebSocket	지속 연결 기반 양방향 통신	브라우저-서버 사이의 양방향 상호작용 세션 제공 	실시간 알림, 진행률 표시, 협업 기능에 유리	단순 CRUD까지 전부 이 방식으로 만들면 구조가 복잡해짐	실시간 기능이 있을 때 높음
MQTT	publish/subscribe	경량 메시징, 제약 장치·저대역폭 환경에 적합 	IoT 센서·스마트펜·임베디드 장치 연동에 강함	일반 모바일 앱 백엔드 API 대체용으로는 과할 수 있음	IoT 확장 시 높음


각 방식의 의미를 프로젝트 관점에서 해석하면

1) REST API

REST API는 클라이언트가 서버에 요청하고, 서버가 응답을 반환하는 기본 구조에 가장 잘 맞습니다.
예를 들어 다음 기능은 REST API가 가장 자연스럽습니다.

회원가입 / 로그인

사용자 정보 조회

필체 이미지 업로드 요청

분석 결과 조회

이전 리포트 목록 조회


즉, 앱의 기본 기능 대부분은 REST API로 처리하는 것이 가장 단순하고 안정적입니다. 

2) gRPC

gRPC는 서버와 서버 사이의 내부 통신에 특히 강합니다.
예를 들어 일반 서버가 딥러닝 서버에 다음과 같이 요청할 수 있습니다.

이미지 분석 요청

획 정보 추출 요청

오차율 리포트 생성 요청


이런 내부 호출은 HTTP 엔드포인트를 두드리는 감각보다, 명확한 서비스 메서드를 호출하는 구조에 가깝습니다. 딥러닝 서버처럼 내부 계약을 엄격하게 관리하고 싶은 계층에 잘 맞습니다. 

3) WebSocket

WebSocket은 실시간성이 필요할 때 의미가 큽니다.
예를 들어 다음 기능이 있다면 WebSocket이 유용합니다.

분석 진행률 실시간 표시

“분석 중 → 완료” 상태 즉시 반영

실시간 협업 필기 교정

교사-학생 간 즉시 피드백


즉, WebSocket은 기본 CRUD API를 대체하는 기술이라기보다, 실시간 이벤트 전달용 채널로 보는 편이 맞습니다. 

4) MQTT

MQTT는 이번 프로젝트의 기본 모바일 앱 백엔드에는 필수는 아닙니다.
하지만 다음처럼 IoT 장치가 붙는 순간 매우 유용해집니다.

스마트펜이 실시간 필기 데이터를 전송

외부 센서가 입력 데이터를 전송

저사양 장치가 서버와 지속 연결

네트워크가 불안정한 환경에서 메시지 전달


따라서 스마트 디바이스 연동형 확장 프로젝트라면 MQTT의 가치가 크게 올라갑니다. 

✅ 이 프로젝트에서의 최종 추천

현재 단계의 가장 현실적인 선택은 다음과 같습니다.

1. 클라이언트 ↔ 일반 서버: REST API


2. 일반 서버 ↔ 딥러닝 서버: gRPC 또는 내부 API


3. 실시간 진행률/상태 알림이 필요할 경우: WebSocket 추가


4. 향후 스마트펜·IoT 장치 연동 시: MQTT 검토



즉, 이 프로젝트의 기본 골격은 REST API 중심으로 두고, 내부 분석 호출은 gRPC, 선택적 실시간 기능은 WebSocket, IoT 확장은 MQTT로 분리하는 방식이 가장 합리적입니다.

최종 결론

이번 필체 교정 앱 프로젝트에서는 다음 구조를 가장 추천합니다.

백엔드 프레임워크: FastAPI

클라이언트와의 기본 통신: REST API

딥러닝 서버와의 내부 통신: gRPC

실시간 기능이 필요할 때: WebSocket 추가

향후 IoT 장치 확장 시: MQTT 검토


그 이유는 다음과 같습니다.

1. FastAPI는 Python 기반이라 AI 처리와 연결이 좋습니다. 


2. OpenAPI 기반 자동 문서화와 요청/응답 검증이 강합니다. 


3. REST API는 모바일 앱의 일반 기능을 구현하기 가장 단순하고 안정적입니다. 


4. gRPC는 내부 AI 서버 호출 구조를 더 명확하고 효율적으로 만듭니다. 


5. WebSocket과 MQTT는 필요할 때 선택적으로 붙이는 것이 맞습니다. 



따라서 현재 단계의 가장 적절한 아키텍처 제안은 다음 한 줄로 요약할 수 있습니다.

> “클라이언트에는 FastAPI 기반 REST API를 제공하고, 내부 AI 분석 계층은 gRPC로 분리하며, 실시간성과 IoT는 필요 시 WebSocket과 MQTT로 확장한다.”
